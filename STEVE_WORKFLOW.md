# Steve의 Gomi 작업 방법

## 평소에는 세 번만 말하면 됩니다

1. **원하는 변경을 말합니다.**
   - Codex가 저장소 상태를 확인하고, 변경을 만들고, 로컬에서 테스트합니다.
   - 화면이 바뀌면 데스크톱과 모바일을 확인합니다.
   - 미리보기와 검증 결과를 먼저 보고합니다.

2. **“ㅇㅋ 커밋”이라고 말합니다.**
   - Codex가 변경 내용을 다시 확인합니다.
   - 필요한 파일만 커밋하고 푸시한 뒤 PR을 준비합니다.
   - Steve는 PowerShell, 파일 경로, 반복적인 GitHub 작업을 직접 할 필요가 없습니다.

3. **“ㅇㅋ 병합”이라고 말합니다.**
   - Codex가 PR 상태를 확인하고 병합과 배포를 진행합니다.
   - 공개된 실제 사이트까지 확인한 뒤 결과를 알려 줍니다.

## 가끔 필요한 한 번의 행동

GitHub 인증이나 권한 승인이 실제로 필요한 경우에만 Steve가 한 번 직접 처리할 수 있습니다. 이때 Codex가 필요한 정확한 한 가지 행동만 안내합니다. 그 밖의 작업은 Codex가 계속 처리합니다.

## 작업이 엉뚱한 곳에서 시작됐을 때

앞으로 Gomi 관련 작업은 Codex에 보이는 **gomisgoldendays 프로젝트 안에서** 시작해야 합니다. 프로젝트가 없는 일반 작업에서 시작하면 이 저장소와 연결되지 않을 수 있습니다.

영구 로컬 폴더:

`C:\Users\user\repos\gomisgoldendays`

이 폴더는 OneDrive 바깥에 두어야 합니다. OneDrive가 동기화하는 위치에 저장소를 두면 `.git` 내부 파일이 클라우드 전용으로 비워지거나 동기화 충돌 사본이 생겨 저장소가 깨질 수 있습니다.

새 작업을 시작할 때 다음 세 가지를 확인합니다.

- 프로젝트 이름이 `gomisgoldendays`로 보이는가?
- `local` 작업으로 표시되는가?
- 현재 브랜치 이름이 보이는가?

세 가지가 모두 보이면 올바른 프로젝트에서 시작한 것입니다.
